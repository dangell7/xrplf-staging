Findings

Seven things worth pulling out from the enumeration. The detail tabs hold the row-level data.

Inherited Hooks / WASM forks

Foundation owns seven forks no one outside Hooks land needs.

llvm-project, wasi-sdk, wasi-libc, wasmi, hook-cleaner-c, hook-macros, Hooks-Homepage. These were inherited when the Foundation absorbed Hooks-era infrastructure. They are not on the mainline development path for amendments and they distort the org's "what does XRPLF actually maintain" story. Decision needed: keep maintained as a Hooks sidechain support kit, transfer to a separate org (e.g. Xahau / Transia), or archive.

Likely-dead niche tooling

A dozen repos haven't been touched in over a year.

Best-guess dormant: xrpl-cli, xrpl-deserializer-c, xrpl-node-configurator, validator_config_creator, xbridge-cli, XRPL-Cluster.com, XRPL-Technical-Explorer, xrpl.ws-stats.com-Frontend, Supply-Calculator, Validation-Stream, rpc2ws, DD-IA.v3. Same playbook as the dead SDKs in A5 — each gets archive vs. revive vs. fund-a-maintainer. None should sit unmarked under the Foundation's brand if it's effectively unmaintained.

Documentation drift

Two doc surfaces, neither is the canonical XRPL spec.

xrpl-dev-portal is the active dev docs site (mirrors xrpl.org), XRPL-Standards hosts the XLS specs, and Understanding_the_XRP_Ledger is a long-form book. Nothing in this set is the formal protocol specification — the closest artefact, Common Prefix's "Common Prefix" spec, lives outside XRPLF entirely. The Documentation Automation SOW (drafted) is the proposed answer to this.

Bus-factor concentration

Critical SDKs all sit at bus-factor 1–2.

From the A5 audit: xrpl-py, xrpl.js, xrpl4j, and now xrpl-rust (newly XRPLF-official) are each held up by a small number of named maintainers — mostly Ripple staff (Py/JS), Peersyst / Vet / XRPLF community (Java / Rust). A7 (PR template rollout) is the lightweight surface fix; the real fix is hiring or sponsoring one or two SDK custodians, which is a 2027 budget conversation.

Recently consolidated

xrpl-rust moved into XRPLF — model the next ones on it.

The sephynox/xrpl-rust → XRPLF/xrpl-rust transfer (confirmed in A5 audit, v1.1.0 released Apr 2026) is the right pattern: take a healthy community SDK, formally adopt it, fund maintenance. Candidates for the same treatment: xrpl-go (Peersyst's Go lib), xrpl4j (already inside but needs explicit ownership). C# / Swift / Ruby remain in the audit-or-leave bucket from the A5 Brett thread.

New Foundation tooling

The release-notifier is live, but the rest of the new tooling layer isn't there yet.

xrplf-release-notifier shipped this week, Mattermost notifications flowing. The implied missing siblings — an amendment status dashboard (C1), a UNL candidate tracker, an automated repo health page — should live in the same org under the same brand. Worth registering placeholders before someone else does.

Meta-repos need owners

.github, actions, ci, docker, packages — who owns these?

Six repos that look like shared org infrastructure (PR templates, reusable workflows, packaging). If A7 (org-wide PR template) is going to ship, .github is where it lives. Confirm who has write access to each before standardising anything, otherwise the rollout will quietly fail for half the repos.
